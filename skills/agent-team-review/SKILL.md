# 多角色协作评审 (Agent Team Review)

> 以"总分总"结构组织多角色独立评审，综合仲裁后输出统一行动计划

## 触发条件

满足以下**任一**条件时激活：

| 条件 | 示例 |
|------|------|
| 评审方案/规范/代码 | "评审这个技术方案"、"Review 这份规范" |
| 多角色/多视角评审 | "从不同角度评审"、"双视角评审" |
| 用户明确要求 | "启动评审"、"用 agent-team-review" |

## 协作骨架：总分总 (Align → Explore → Merge)

```
┌─────────────────────────────────┐
│  1. Align（对齐）                │
│  - 确认评审对象、范围、成功标准  │
│  - 选定角色组合与关注优先级      │
├─────────────────────────────────┤
│  2. Explore（独立评审）          │
│  各角色独立输出评审章节          │
│  角色之间禁止互相引用            │
├─────────────────────────────────┤
│  3. Merge（综合仲裁）            │
│  - 合并去重、冲突仲裁            │
│  - 优先级排序、统一行动计划      │
└─────────────────────────────────┘
```

## 默认角色（3 角色）

评审默认使用以下 3 个角色，用户可根据场景增减（上限 5 个）。

### 角色 A：结构与一致性评审员

- **Identity**: 你是结构与一致性评审员
- **Scope**: 目录结构、命名规范、入口文件完整性、引用一致性、Single Source of Truth
- **Anti-patterns**: 不评价业务逻辑正确性，不讨论实现细节

### 角色 B：可执行性评审员

- **Identity**: 你是可执行性评审员
- **Scope**: 行动项可落地性、验收标准明确性、依赖识别、优先级合理性、MVP 边界
- **Anti-patterns**: 不纠结格式细节，不做架构重设计

### 角色 C：风险与边界分析师

- **Identity**: 你是风险与边界分析师
- **Scope**: 安全风险、范围漂移、过度工程化、向后兼容、依赖风险、滥用可能
- **Anti-patterns**: 不提改进方案（那是行动项的事），只识别风险和边界问题

### 自定义角色

用户可替换或新增角色，每个角色定义须包含：

| 字段 | 说明 |
|------|------|
| Identity | 角色身份（一句话） |
| Scope | 只关注什么 |
| Anti-patterns | 应避免什么 |

## 输出契约

所有评审输出**必须**包含以下基础契约字段：

| 字段 | 说明 | 必需 |
|------|------|------|
| **TL;DR** | 结论摘要，<= 3 句 | 是 |
| **Findings** | 客观发现，每项带证据/引用。按 P0/P1/P2 分级 | 是 |
| **Risks** | 未来影响预判（概率/严重度/缓解建议） | 是 |
| **Actions** | 行动项（Owner/优先级/依赖/验收标准） | 是 |
| **Open Questions** | 未决问题（阻塞项、需补材料） | 按需 |

### Findings 分级标准

| 级别 | 含义 | 判断标准 |
|------|------|---------|
| **P0** | 阻塞级 | 安全问题、核心逻辑错误、阻塞后续工作 |
| **P1** | 重要 | 功能缺陷、设计不一致、用户体验显著影响 |
| **P2** | 改善 | 规范对齐、体验优化、可延后处理 |

### 评审扩展字段（可选）

| 字段 | 说明 |
|------|------|
| Migration Notes | 从旧版迁移的步骤 |
| Acceptance Criteria | 怎么判断改好了 |
| 综合评分 | 按维度打分（1-5） |

## 单 Agent 实现策略

当前阶段采用**串行角色切换**（单次会话中依次以不同角色输出）：

### 执行流程

1. **Align 阶段**：探测上下文 + 确认评审对象、范围、角色选择
   - **AI-TASK 项目探测**：检查当前目录是否存在 `ai-task/` 目录（软链接或真实目录）
     - 若存在 → 读取以下文件，作为评审背景上下文：
       1. `ai-task/project.yaml` — 项目代号、技术栈
       2. `ai-task/index.md` — 任务列表、规范声明
       3. `ai-task/README.md` — **任务命名规范、文件夹结构规则（落盘时必须遵守）**
     - 若不存在 → 跳过，按通用模式执行评审
   - 确认评审对象、范围、角色选择
2. **Explore 阶段**：依次以每个角色身份输出独立评审章节
   - 每个角色输出以 `## Agent {X} 评审报告：{角色名}` 为标题
   - 角色之间**禁止互相引用**（在 Merge 阶段统一引用）
3. **Merge 阶段**：综合所有角色输出
   - 提取共识点
   - 冲突仲裁
   - 输出统一行动计划

### 文件落盘规则（AI-TASK 项目）

评审产出需写入 ai-task 任务目录时，**必须**遵守以下命名规范（优先级高于自由命名）：

| 场景 | 规则 |
|------|------|
| 文件夹任务的入口文件 | **必须**命名为 `task_plan.md`，不得使用其他名称 |
| 子文件命名格式 | `task_{N}_{角色/主题}.md`，编号从 1 递增 |
| 评审主文档（Merge 产出） | 写入 `task_plan.md` |
| 各角色独立报告 | 写入 `task_1_{角色A}.md`、`task_2_{角色B}.md`… |
| 任务目录编号 | 查看 `tasks/` 目录当天日期下最大编号 +1，**日内递增** |

**常见错误**：
- ❌ `20260303-002_评审主文档.md`（应为 `task_plan.md`）
- ❌ `AAR改动实现规格.md`（应为 `task_1_AAR改动规格.md`）
- ✅ 文件夹 `20260303-002_[评审]xxx/` + 内部 `task_plan.md` + `task_1_xxx.md`

### 角色隔离保障

- 每个角色 prompt 包含 Identity / Scope / Anti-patterns
- 角色输出以独立章节分隔
- 评估指标：各角色 Findings 去重后独立发现占比 >= 50%

## 合并规则

| 场景 | 处理方式 |
|------|---------|
| 多角色发现重叠 | 保留证据最充分的版本 |
| 优先级冲突 | 按影响范围裁决：P0（阻塞/安全）> P1（核心功能）> P2（体验/规范） |
| 行动项合并 | 去重后按 Owner 分组 |
| 评分差异 | 取加权平均，附注分歧理由 |
